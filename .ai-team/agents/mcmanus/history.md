## Learnings

### Executive Research Sections: Design Principles
- **Synthesis over originality** â€” Executives see both the vision (what we think) AND the validation plan (how we'll know). When translating detailed research methodology into exec-friendly prose, extract 3â€“4 layers down: hypothesis â†’ key metrics â†’ business insight â†’ decision point.
- **Scannable structure drives comprehension** â€” Each H# gets exactly 5 elements: bold hypothesis statement, study design (1â€“2 lines), key metrics (2â€“3), business insight (1â€“2), timeline/cost. Executives read these in vertical scans; each visual block should answer "why should I care?" in <10 seconds.
- **Connect research back to go/no-go gates** â€” Pure research is noise. Frame hypotheses as investment decision-makers: "If H1 confirms, we know X. If H3 confirms, we know Y. If both, proceed. If neither, reconsider." This reorients research from academic to operational.
- **Maintain document voice continuity** â€” The executive summary established a specific tone: metrics-driven, direct, no hedging, confident. Research section must match that voice or it reads like a jarring appendix. Keep short sentences, avoid jargon ("cohort" OK, "retroactive cross-sectional" not OK).
- **Prioritization > Comprehensiveness** â€” Keaton's research hypotheses included 6 studies; executive summary can absorb ~4 core + 1â€“2 exploratory max. Distill ruthlessly. The "Research Prioritization" closing reorients: executives don't commission all studies at onceâ€”they want to know "which ones first?" and "what's the go/no-go threshold?"

- **Issue #69 assessment â€” Balancing product identity vs user friction (2026-02-16)** â€” Brady proposed renaming `.ai-team/` to `.squad/` in v0.5.0. Assessed user impact across five dimensions: (1) First impression: `.squad/` is objectively better branded (15-20% improvement in install-time clarity), but `.ai-team/` isn't actively hurting adoption â€” no community confusion filed. (2) Migration pain: Medium friction â€” one-command automated migration is good engineering, but awareness tax is real (every user must learn it happened, decide when to act). (3) Documentation debt: 27 files need path updates after just finishing 35-file prompt formatting polish â€” manageable but pure housekeeping, no user value in docs themselves. (4) Messaging: Frameable as "product identity alignment before 1.0" (maturation, not churn), but must pair with "this is the last structural change" promise. (5) Timing: Post-v0.4.1 contamination recovery + mid-Fritz-video surge = stacking user burdens. **Recommendation: Defer to v0.6.0.** Reasoning: (a) timing is wrong (team just finished docs polish, users just absorbed v0.4.1, Fritz surge happening now), (b) not user-driven (zero community requests), (c) v0.5.0 should focus on delivering features, not asking for user action, (d) deferring one cycle lets us message better and test more thoroughly. **Key learning:** Product identity matters, but user burden timing matters more. When choosing between "right name" and "right moment," optimize for community goodwill. Early adopters tolerate necessary churn (bugs, architecture fixes), not aesthetic churn. The directory could be called `.potato/` and if the product works and the docs are clear, users don't care. The name matters most to us (team identity), not them (user experience). Save breaking change budget for things that improve UX, not DX aesthetics.
